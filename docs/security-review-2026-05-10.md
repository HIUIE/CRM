# SmartTrade AI CRM 安全审计记录

审计日期：2026-05-10

## 审计范围

- 后端鉴权与数据归属隔离：订单、客户、合作伙伴、联系人、AI 分析、批量操作。
- 前端业务功能同步：负责人筛选、首页工作台、订单详情留痕与单据槽位。
- 依赖与构建验证：`npm audit`、TypeScript/ESLint、生产构建、自动化测试。

## 已修复问题

### P1：AI 订单分析接口缺少订单归属校验

`POST /api/ai/analyze-order` 原先会先读取订单详情再交给 AI 分析。若员工知道其他订单号，存在越权分析客户订单资料的风险。

修复结果：

- 调用 `buildOrderDetail` 前先执行 `checkOrderAccess`。
- 无权限时返回 `ORDER_NOT_FOUND`，不泄露订单是否真实存在。
- 新增回归测试：员工无法分析其他负责人订单。

### P1：订单批量状态更新缺少逐单权限校验

`PATCH /api/orders/batch` 原先只校验登录态，未对 `ids` 中每个订单执行归属检查。

修复结果：

- 统一清洗并去重订单 id。
- 校验目标状态必须属于合法订单状态。
- 对每个订单执行 `checkOrderAccess`，任一越权即拒绝批量更新。
- 新增回归测试：员工无法批量修改其他负责人订单。

### P2：合作伙伴与联系人接口缺少编辑归属校验

合作伙伴资料和联系人编辑、删除接口原先存在按 id 直接修改的风险。

修复结果：

- 新增合作伙伴访问校验。
- 编辑合作伙伴、创建联系人、编辑联系人、删除联系人前统一确认当前用户是否有权访问该合作伙伴。
- 合作伙伴详情过滤软删除记录。

## 当前安全状态

- 依赖审计：`npm audit --audit-level=moderate` 未发现中高危依赖漏洞。
- 鉴权框架：登录 Cookie 使用 HttpOnly，状态变更接口启用 CSRF double-submit 校验。
- 敏感配置：AI API Key 等配置支持 AES-256-GCM 加密存储。
- 数据隔离：核心客户、订单、AI 分析、批量操作、合作伙伴资料已纳入服务端权限校验。

## 建议继续加固

1. 为附件下载和预览补充更多端到端权限测试，特别是订单转交后的历史附件访问。
2. 为“负责人转交”建立专门的审计报表，展示转交前后客户、订单、任务是否全部同步成功。
3. 生产环境强制启用 HTTPS，并设置 `COOKIE_SECURE=true`。
4. 为管理员账号增加二次验证或登录 IP 异常提醒。
5. 将系统迁移包与业务归档包的导出、下载、解密操作全部写入审计日志。
6. 定期执行 `npm audit`、数据库备份恢复演练和权限回归测试。
